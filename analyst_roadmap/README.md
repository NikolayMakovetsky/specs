## Концепция документа

Данный документ представляет собой структурированный план по трансформации из начинающего специалиста в системного аналитика, готового к решению бизнес-задач. В качестве метафоры используется структура технического задания, где вместо требований к ПО описываются требования к собственным знаниям, а вместо этапов разработки — этапы обучения.

Документ является **публичным артефактом** и размещён на GitHub для демонстрации подхода к самоорганизации и профессиональному развитию.

---

## Введение

Я — специалист без коммерческого опыта в системном анализе, но с системным мышлением и техническим бэкграундом. Основная проблема заключается в том, что наличие теоретических знаний не подтверждено практическими артефактами, которые можно было бы продемонстрировать потенциальному работодателю.

Данный план призван устранить этот разрыв: в процессе его выполнения я создам портфолио из документов, диаграмм и проектных артефактов, которые наглядно продемонстрируют мои компетенции.

## Глоссарий

| Термин | Определение |
|--------|-------------|
| **Системный аналитик** | Проектировщик цифровых решений, который через активное взаимодействие со всеми стейкхолдерами выявляет истинные потребности бизнеса, документирует их в виде комплексного технического задания и сопровождает этот документ до релиза, гарантируя соответствие разработанной системы утверждённой бизнес-логике и критериям качества. |
| **Техническое задание (ТЗ)** | Основной артефакт работы аналитика — структурированный документ, содержащий бизнес-контекст, функциональные и нефункциональные требования, модели данных, спецификации API и другие сведения, необходимые команде разработки для реализации продукта. |
| **MVP навыков** | Минимальный набор компетенций, достаточный для успешного выполнения задач начинающего системного аналитика: владение нотациями (BPMN, UML), базовое знание SQL, понимание REST API и принципов работы с Git. |
| **Стейкхолдер** | Заинтересованное лицо, которое может влиять на проект или находится под его влиянием. В контексте данного документа — все стороны, вовлечённые в процесс моего профессионального развития и трудоустройства. |
| **Артефакт** | Материальный результат работы аналитика: документ, диаграмма, спецификация, схема данных, макет интерфейса и т.п. |
| **Бизнес-процесс** | Последовательность действий, направленных на достижение конкретной бизнес-цели. Описывается в нотациях (BPMN, EPC) для последующей автоматизации. |

## Цель и задачи

**Цель:** Создать портфолио, демонстрирующее компетенции системного аналитика, и получить не менее **3 приглашений на собеседование** в течение 3 месяцев после публикации резюме.

**Задачи и критерии их выполнения:**

| № | Задача | Измеримый результат (артефакт) |
|---|--------|--------------------------------|
| 1 | Освоить нотацию BPMN для описания бизнес-процессов | Создана минимум **1 диаграмма BPMN** для курсового проекта (основной процесс) |
| 2 | Освоить диаграммы вариантов использования (UML Use Case) | Создана **1 диаграмма Use Case** с минимум 3 актёрами и 5 вариантами использования |
| 3 | Освоить диаграммы последовательности (UML Sequence) | Создана **1 диаграмма последовательности** для ключевого сценария курсового проекта |
| 4 | Освоить проектирование реляционных баз данных и SQL | Создана **ER-диаграмма** (логический уровень) с минимум 6 сущностями и связями. Написано **3 SQL-запроса** разной сложности (SELECT с JOIN, агрегация, вложенный запрос) |
| 5 | Научиться проектировать REST API | Создана **API-спецификация** в формате OpenAPI/YAML или Markdown-таблица с описанием минимум **5 эндпоинтов** (методы, параметры, примеры запросов/ответов) |
| 6 | Изучить принципы асинхронного взаимодействия (очереди сообщений) | В курсовой проект добавлен **текстовый раздел** с обоснованием использования очереди (RabbitMQ/Kafka) для одного из процессов |
| 7 | Разработать полноценный курсовой проект | Написан **документ «Техническая спецификация»**, содержащий все 9 разделов (структура описана ниже) |
| 8 | Подготовить резюме и сопроводительные материалы | Создано **резюме** на hh.ru, оформлен **GitHub-репозиторий** с данным планом и ссылка на него добавлена в резюме |
| 9 | Создать и опубликовать «Библиотеку системного аналитика» (конспекты уроков) | Оформлено **20 конспектов** (по одному на каждый урок) с выделением ключевых понятий, тезисов и связью с курсовым проектом |

## Структура Технической спецификации (курсовой проект)

Техническая спецификация — это главный артефакт курсового проекта. Она состоит из **9 разделов**, каждый из которых отражает один из аспектов работы системного аналитика:

| № раздела | Название раздела | Что входит | Связь с уроками |
|-----------|------------------|------------|-----------------|
| **2.1** | **Перечень бизнес-процессов** | Список всех ключевых процессов проекта с указанием владельцев и триггеров | Уроки 3–4 |
| **2.2** | **Основной бизнес-процесс в нотации BPMN** | Графическая диаграмма с пулами, дорожками, событиями и шлюзами | Урок 4 |
| **2.3** | **Диаграмма вариантов использования (Use Case)** | Диаграмма с актёрами и вариантами использования + описание системы | Урок 5 |
| **2.4** | **Сценарии вариантов использования** | Текстовое описание основного и альтернативного потоков для 1–2 ключевых сценариев | Урок 5 |
| **2.5** | **Функциональные и нефункциональные требования** | Список F-требований (что система должна делать) и NFR-требований (производительность, безопасность и т.д.) | Уроки 5–6 |
| **2.6** | **Модель данных (ER-диаграмма)** | Логическая модель данных: сущности, атрибуты, типы связей, первичные/внешние ключи | Уроки 8–9 |
| **2.7** | **Диаграммы последовательности (Sequence)** | Диаграмма взаимодействия объектов во времени для ключевого сценария | Урок 14 |
| **2.8** | **Описание методов API** | Спецификация REST API: эндпоинты, методы, параметры, примеры запросов/ответов, коды ошибок | Уроки 12, 15 |
| **2.9** | **Декомпозиция задач для разработки** | Разбивка работы на эпики, истории и задачи с оценкой трудоёмкости | Урок 19 |

## Стейкхолдеры

В контексте данного документа выделены три ключевые заинтересованные стороны. Для каждой указаны зона ответственности и критерии успеха.

| Стейкхолдер | Роль в проекте | Ключевой интерес | Критерий успеха (с точки зрения этого стейкхолдера) |
|-------------|----------------|-------------------|------------------------------------------------------|
| **Я (Исполнитель)** | Разрабатываю план, учусь, создаю артефакты | Получить работу и профессионально реализоваться | Документ выполнен, артефакты созданы, есть оффер |
| **Рекрутер / HR** | Первичный фильтр: оценивает резюме и портфолио | Найти специалиста, который подойдёт команде и не «прогорит» на испытательном сроке | Резюме содержит понятный стек, портфолио демонстрирует системный подход и реальные артефакты, а не просто слова |
| **Будущий Team Lead** | Вторичный фильтр: техническое интервью, оценка хардов | Убедиться, что кандидат способен самостоятельно вести задачи, понимает архитектуру и не требует постоянного контроля | Кандидат отвечает на вопросы по нотациям, SQL, API; может аргументировать свои проектные решения |

> **[Здесь будет вставлена матрица влияния]**

*Описание матрицы:*
- **Team Lead** — высокое влияние + высокий интерес (ключевой стейкхолдер)
- **Я (Исполнитель)** — высокое влияние + высокий интерес (исполнитель и бенефициар)

## Функциональные требования к навыкам

Системный аналитик должен владеть следующим набором компетенций (разбиты по уровню освоения к концу проекта):

| Категория | Навык | Уровень к концу проекта |
|-----------|-------|--------------------------|
| **Бизнес-анализ** | Выявление требований через интервью, воркшопы, анализ документации | Базовый (теория + 1 практика) |
| **Документирование** | Составление Vision-документа, Технической спецификации (ТЗ) | Продвинутый (полноценный документ) |
| **Нотации** | BPMN (моделирование бизнес-процессов) | Практический (диаграмма для проекта) |
| **Нотации** | UML (Use Case, Sequence, классов) | Практический (Use Case и Sequence) |
| **Базы данных** | Проектирование ER-диаграмм, написание SQL-запросов (SELECT, JOIN, GROUP BY) | Базовый-практический (модель + 3 запроса) |
| **API** | Проектирование REST API (методы, URL, JSON, коды ответов) | Практический (спецификация из 5+ эндпоинтов) |
| **Архитектура** | Понимание монолитной и микросервисной архитектур | Теоретический (знание плюсов/минусов) |
| **Интеграции** | Принципы синхронного и асинхронного взаимодействия (RabbitMQ/Kafka) | Базовый (понимание, когда что применять) |
| **Безопасность** | Аутентификация, авторизация, JWT, OAuth 2.0 | Базовый (понимание принципов) |
| **Инструменты** | Git, Markdown, Draw.io, Postman/Swagger | Практический (все используются в проекте) |
| **Тестирование** | Составление тест-кейсов, понимание процесса тестирования | Теоретический (понимание) |

## Нефункциональные требования

Ограничения, в рамках которых реализуется данный план:

| Категория | Требование | Обоснование |
|-----------|------------|-------------|
| **Временные** | Общая продолжительность — **3 месяца** (12 недель) | Реалистичный срок для совмещения с основной деятельностью (поиск работы не является единственным занятием) |
| **Бюджетные** | 0 ₽ (используются только бесплатные инструменты и ресурсы) | Нет финансовых вложений, только время и усилия |
| **Ресурсные** | Доступны: компьютер, интернет, бесплатные версии ПО (Draw.io, Git, PostgreSQL, Postman) | Нет необходимости покупать лицензии |
| **Качественные (к документу)** | Все артефакты должны быть понятны техническому специалисту с опытом от 2 лет (минимальный уровень) | Документы не должны требовать дополнительных устных пояснений — они «говорят сами за себя» |
| **Качественные (к обучению)** | Каждый изученный урок должен иметь практическое отражение в курсовом проекте | Исключается «пассивное» обучение — вся теория проверяется на практике |
| **Приёмочные** | GitHub-репозиторий должен быть открытым и содержать README с навигацией | Портфолио должно быть доступно для просмотра без запроса доступа |

## Критерии приёмки

Проект считается выполненным успешно при соблюдении **всех** следующих условий:
