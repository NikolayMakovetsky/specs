# Конспект лекции №2: Введение в профессию системного аналитика

---

## 1. Организационная информация

- **Формат курса:** 16 офлайн-занятий + 8 онлайн-вебинаров
- **Учебный проект:** добровольный, с куратором
- **Сроки:** 2 недели после окончания курса на завершение проекта
- **Итоги:** вручение сертификатов, вечеринка

### Структура курса
- **Требования** — большой блок
- **Архитектура** — большой блок
- **Базы данных**
- **Инструменты аналитика**

---

## 2. Кто такой системный аналитик?

### Бизнес-аналитик vs Системный аналитик

| | Бизнес-аналитик | Системный аналитик |
|---|-----------------|-------------------|
| **Что знает** | Знает, что хотят пользователи | Знает, как реализовать через систему |
| **С кем работает** | С заказчиком, бизнесом | С командой разработки |
| **Задача** | Собирает бизнес-требования | Переводит требования в техзадание |

> **На практике** границы размыты. Чаще всего это **«человек-оркестр»** — один специалист совмещает обе роли.

### Определение
> **Системный аналитик** — специалист, который снимает проблемы, переводит их в задачи для проектной команды и определяет, что необходимо разработать.

---

## 3. Задачи системного аналитика (классификация)

| Тип задачи | Пример |
|------------|--------|
| **Небольшие** | Анализ существующих систем, выбор оптимальной под требования заказчика |
| **Средние** | Описание бизнес-требований для нового функционала |
| **Крупные** | Разработка ТЗ на приложение с нуля (напр., система лояльности для ТРЦ) |

---

## 4. Программное обеспечение

### Виды ПО
1. **Готовые решения (коробочные)** — существуют на рынке
2. **Кастомизированные** — готовый продукт дорабатывается под заказчика
3. **Индивидуальные** — разработка с нуля под конкретный проект

**Связь с задачами аналитика:**
- Готовое ПО → небольшие задачи (обзор, выбор)
- Кастомизированное → средние/крупные задачи
- Индивидуальное → всегда крупная задача

---

## 5. Команда разработки ПО

### Полный состав проектной команды
1. **Project Manager** — управление сроками, координация
2. **Архитектор** — проектирование архитектуры
3. **Системный аналитик** — сбор требований, ТЗ
4. **UX/UI Дизайнер** — интерфейсы
5. **Backend-разработчик** — серверная часть
6. **Frontend-разработчик** — клиентская часть
7. **Тестировщик (QA)** — проверка качества

### Минимальный состав
- Project Manager
- Системный аналитик
- Дизайнер
- Разработчик

### Дополнительные роли
- Маркетолог
- Техподдержка
- Юристы (для регулируемых отраслей)
- Product Manager

---

## 6. Результаты работы системного аналитика

1. **Техническое задание (ТЗ)** — основной документ
2. **Руководство пользователя** (если нет технического писателя)

### Взаимодействие аналитика
- **С заказчиком** — сбор требований
- **С разработчиками** — уточнение логики, ответы на вопросы
- **С тестировщиками** — составление тестов, анализ ошибок

---

## 7. Навыки (Soft Skills vs Hard Skills)

### Soft Skills (универсальные навыки)
- Коммуникативные навыки
- Контроль процессов
- Системное мышление
- Грамотная речь
- Перфекционизм (внимательность)
- Хорошая память

### Hard Skills (профессиональные)
- Диаграммы (UML, BPMN и др.)
- Языки запросов к БД (SQL)
- Знание аббревиатур и терминологии

---

## 8. Жизненный цикл ПО

### Этапы жизненного цикла
1. **Анализ требований**
2. **Проектирование**
3. **Программирование**
4. **Тестирование и отладка**
5. **Эксплуатация и сопровождение**

> Системный аналитик участвует на **всех этапах**, но ключевая роль — этапы 1 и 2.

**Важно:** «Нулевой этап» — возникновение идеи — не входит в классический жизненный цикл.

---

## 9. Модели управления разработкой

### Водопад (каскадная модель)

| Период | Модель | Особенность |
|--------|--------|-------------|
| 1970–1985 | Классический водопад | Этапы строго последовательно |
| 1985–1990 | Итеративная модель | Возможен возврат к предыдущим этапам |
| 1990+ | Циклическая (спиральная) | Каждая итерация расширяет проект |

**Плюсы водопада:**
- Понятно, что получится
- Понятен бюджет
- Легко управлять

**Минусы водопада:**
- Ошибки архитектуры обнаруживаются поздно
- Долгая разработка финального продукта

---

### Гибкие методологии (RAD — Rapid Application Development)

**История:** появились в 1990-х для малых коммерческих компаний.

**Суть:** быстрый цикл «требование → анализ → проектирование → разработка → внедрение»

**Плюсы:**
- Ускоренная разработка
- Быстрая доставка продукта
- Высокая адаптивность (можно менять требования)

**Минусы:**
- Требуется высокая квалификация команды
- Нужны сильные организаторские навыки PM

**Примеры гибких методологий:**
- RAD (1991)
- MSF (1994)
- Scrum, XP, Kanban и др.

---

## 10. Какую методологию выбрать?

| Ситуация | Рекомендуемая модель |
|----------|---------------------|
| Стартап, малый бюджет, неизвестно, выстрелит ли идея | **Гибкая (RAD/Scrum)** |
| Крупный клиент, долгосрочный проект (1-1.5 года) | **Водопад + элементы гибких** |
| Продукт может потерять актуальность через 2-3 месяца | **Гибкая** |

## 11. Вопросы из аудитории (FAQ)

> **Кто пишет техническое задание, если нет архитектора?**

- Системный аналитик, но с привлечением опытных разработчиков/техлида.

> **Аналитик считает часы на разработку?**

- Да, приблизительно. Точная оценка — после консультации с архитектором.

> **Бывает, что разработка уже началась, а требования ещё правятся?**

- Да, часто. Это корректируется в процессе (особенно при гибких методологиях).

> **Может ли аналитик писать ТЗ без архитектора?**

- Да, но сложные технические решения лучше согласовывать с разработчиками.

- **Команда:** 100–150 человек, ~15 команд
- **Роли:** Продукт-оунеры, архитекторы (техлиды), разработчики, тестировщики
- **Особенность:** система разработана с нуля внутри компании, не покупная

## 13. Ключевые тезисы (шпаргалка)

1. **Системный аналитик** — связующее звено между бизнесом и разработкой.
2. На практике часто совмещает роли **бизнес-аналитика**.
3. Основной результат — **техническое задание**.
4. Участвует во **всех этапах** жизненного цикла ПО.
5. Модели управления: **водопад** (для крупных проектов) и **гибкие** (для быстрых).
6. Главные навыки: **коммуникация + системное мышление**.
