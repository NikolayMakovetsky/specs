# Конспект лекции №13: Монолитная и микросервисная архитектура

---

## 1. Введение: что такое backend-приложение

**Backend** — это программа, которая запускается на сервере и обрабатывает запросы от клиентской части (фронтенда).

**Как это работает:**
1. Разработчики пишут код на языке программирования
2. Код компилируется в исполняемый файл (бинарник, .exe)
3. Исполняемый файл загружается на сервер и запускается
4. Программа слушает входящие запросы, обрабатывает их, обращается к БД, возвращает ответы

---

## 2. Монолитная архитектура

**Монолит** — это одно большое приложение, в котором содержится вся логика системы.

### Структура монолита:
- Единая кодовая база
- Всё в одном исполняемом файле
- Одна база данных для всего приложения
- Вся логика: представление, бизнес-логика, доступ к данным, безопасность

### Компоненты монолита:
- **Балансировщик нагрузки** (Load Balancer) — распределяет запросы между несколькими копиями приложения
- **Несколько экземпляров** — запускаются для масштабирования

### Преимущества монолита:
| Преимущество | Описание |
|--------------|----------|
| **Быстрый старт** | Проще и быстрее разработать MVP |
| **Простое тестирование** | Одно приложение — один тест |
| **Простой деплой** | Один исполняемый файл — один запуск |
| **Простое масштабирование** | Запуск нескольких копий через балансировщик |

### Недостатки монолита:
| Недостаток | Описание |
|------------|----------|
| **Огромный код** | Сложно ориентироваться, добавлять новый функционал |
| **Холостое масштабирование** | При росте нагрузки на один компонент (например, заказы) — приходится запускать **полную копию** всего приложения |
| **Отсутствие изоляции** | Ошибка в одном компоненте может сломать другой |

### Схема монолита:
```
[Клиент] → [Балансировщик] → [Монолит (экз.1)]
                          → [Монолит (экз.2)]
                          → [Монолит (экз.3)]
                                    ↓
                               [Общая БД]
```

## 3. Микросервисная архитектура

**Микросервисы** — архитектура, в которой приложение разбито на множество маленьких независимых служб (сервисов), каждая из которых отвечает за свою бизнес-функцию.

### Основные характеристики:
- Каждый микросервис имеет **свою базу данных**
- Микросервисы общаются друг с другом через **API** (обычно REST)
- Микросервисы могут быть написаны на **разных языках программирования**
- **Горизонтальное масштабирование** каждого сервиса независимо

### Компоненты микросервисной архитектуры:
| Компонент | Описание |
|-----------|----------|
| **Микросервисы** | Независимые службы, каждая со своей БД |
| **API Gateway** | Единая точка входа для клиентов, маршрутизация запросов |
| **Service Registry** | Хранит информацию о том, где запущены сервисы |
| **Балансировщик** | Распределение нагрузки между экземплярами |
| **Централизованная конфигурация** | Управление настройками всех сервисов |
| **Контейнеризация** | Каждый сервис в отдельном контейнере (Docker) |
| **Оркестрация** | Управление контейнерами (Kubernetes) |
| **CI/CD** | Непрерывная интеграция и развёртывание |

### Преимущества микросервисов:
| Преимущество | Описание |
|--------------|----------|
| **Технологическая гибкость** | Каждый сервис на своём языке/стеке |
| **Децентрализованное управление данными** | Каждый сервис владеет своей БД |
| **Независимое развёртывание** | Можно обновить один сервис без остановки других |
| **Гибкое масштабирование** | Масштабируем только те сервисы, которые нуждаются в этом |

### Недостатки микросервисов:
| Недостаток | Описание |
|------------|----------|
| **Высокий порог вхождения** | Сложно проектировать с нуля, нужно много инфраструктуры |
| **Сложность взаимодействия** | Много сетевых вызовов, возможны задержки |
| **Транзакционность** | Сложно обеспечить атомарность операций между сервисами |
| **Нарушение DRY** | Возможно дублирование кода в разных сервисах |

## 4. Транзакционность в микросервисах

**Проблема:** при выполнении операции, затрагивающей несколько микросервисов, один из них может упасть. Данные окажутся несогласованными.

**Пример:** клиент оплатил заказ (деньги списаны), но статус заказа не обновился.

**Требование:** операция должна быть либо полностью выполнена, либо полностью откатана (в исходное состояние).

**Решение:**
- Использовать **Saga-паттерн** (последовательность локальных транзакций)
- Повторные попытки (Retry)
- Компенсирующие операции (откат)

---

## 5. Graceful Shutdown (бесшовное обновление)

**Проблема:** при обновлении сервиса нельзя просто остановить старую версию, если она обрабатывает запросы.

**Процесс:**
1. Запускаются **две новые версии** сервиса
2. Балансировщик переключает **весь трафик** на новые версии
3. Старые версии **дожидаются завершения текущих запросов**
4. Затем старые версии **останавливаются**

> **Клиент не замечает переключения** — система продолжает работать непрерывно.

## 6. Stateful vs Stateless

| Тип | Описание | Особенность |
|-----|----------|-------------|
| **Stateful** | Хранит состояние сессии внутри себя | Прибить нельзя без потери данных |
| **Stateless** | Не хранит состояние, всё в БД | Можно убивать и перезапускать в любой момент |

**Рекомендация:** все микросервисы должны быть **stateless** — это упрощает масштабирование и обновление.

## 7. Практический пример: разбиение монолита на микросервисы

### Исходные сущности:
1. **Пользователь** (авторизация, роли)
2. **Заявки** (создание, получение, статусы)
3. **Соискатели** (кандидаты)
4. **Работодатели** (компании)
5. **Подбор** (маппинг заявок на соискателей)
6. **Уведомления** (email, SMS, Telegram)

### Сервисы:
| Микросервис | База данных | Функция |
|-------------|-------------|---------|
| **Сервис пользователей** | Своя БД | Авторизация, роли, управление пользователями |
| **Сервис заявок** | Своя БД | CRUD заявок, статусы |
| **Сервис соискателей** | Своя БД | Хранение данных соискателей |
| **Сервис работодателей** | Своя БД | Хранение данных работодателей |
| **Сервис подбора (Matcher)** | Своя БД | Поиск соответствия заявка ↔ соискатель |
| **Сервис уведомлений** | Своя БД | Отправка уведомлений (email, SMS, Telegram) |
| **Адаптеры** | Нет БД | Интеграция с внешними сервисами (почта, SMS-шлюз, Telegram) |

### Взаимодействие сервисов:
```
[Клиент (фронтенд)]
        ↓
[API Gateway] → [Сервис заявок] → [Сервис подбора] → [Сервис уведомлений] → [Почта/SMS/Telegram]
        ↓         ↓                ↓
[Сервис пользователей] ←→ [БД]
[Сервис соискателей]   ←→ [БД]
[Сервис работодателей] ←→ [БД]
```

## 8. Сервис-ориентированная архитектура (SOA)

**SOA (Service-Oriented Architecture)** — архитектура, в которой сервисы общаются через **общую шину данных** и используют **общую БД**.

**Особенности:**
- Единая база данных для всех сервисов
- Общение через шину (Enterprise Service Bus)
- Сейчас считается устаревшей, но ещё встречается

**Разница с микросервисами:** в микросервисах — у каждого своя БД; в SOA — общая БД.

---

## 9. Итоговые тезисы (шпаргалка)

1. **Монолит** — одно большое приложение, проще для старта, но сложно масштабировать
2. **Микросервисы** — много маленьких сервисов, каждый со своей БД, гибко, но сложно
3. В микросервисах — **горизонтальное масштабирование** по потребностям
4. **Stateless** — сервис не хранит состояние, упрощает обновление
5. **Graceful shutdown** — бесшовное обновление без потери запросов
6. **Транзакционность** — сложная проблема в микросервисах
7. На практике часто используют **гибридную архитектуру** — что-то среднее между SOA и микросервисами
8. Аналитик должен понимать архитектуру, чтобы:
   - Проектировать новые функции с учётом существующих сервисов
   - Понимать, как данные распределены между сервисами
   - Общаться с разработчиками на одном языке
