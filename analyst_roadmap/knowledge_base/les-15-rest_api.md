# Конспект лекции №15: Проектирование REST API

---

## 1. Что такое API?

**API (Application Programming Interface)** — это **контракт** между системами.

- Сервер/сервис предоставляет API и говорит: «Ты можешь обращаться ко мне так, а я обязуюсь отвечать так».
- API описывает:
  - Какие операции доступны
  - Что нужно передать на вход
  - Что придёт на выход

**Пример:** вы ищете авиабилеты — вводите город, дату → отправляете запрос → получаете список рейсов.

---

## 2. Виды интеграций

| Тип | Описание | Пример |
|-----|----------|--------|
| **Общая БД** | Два приложения работают с одной БД | Сервис A создаёт запись, сервис B читает её по расписанию |
| **Обмен файлами** | Запись/чтение из файлового хранилища | Одно приложение кладёт файл, другое читает |
| **Удалённые вызовы (синхронные)** | Клиент ждёт ответа | REST API, SOAP, gRPC |
| **Обмен сообщениями (асинхронные)** | Отправка без ожидания ответа | Очереди, брокеры (Kafka, RabbitMQ) |

**Важно:** общая БД и файловый обмен — не лучшие практики, но иногда встречаются.

---

## 3. Синхронное vs Асинхронное взаимодействие

| Характеристика | Синхронное | Асинхронное |
|----------------|------------|-------------|
| **Ожидание ответа** | Да (клиент блокируется) | Нет |
| **Пример** | Запрос к API, получение результата | Отправка в очередь, обработка позже |
| **Когда использовать** | Когда нужен результат немедленно | Для рассылок, тяжёлых операций |

---

## 4. Что такое REST?

**REST (Representational State Transfer)** — это **архитектурный стиль** (паттерн), а не протокол.

> REST ≠ HTTP, но чаще всего REST-API реализуется через HTTP.

**Принципы REST:**

1. **Клиент-серверная архитектура** — разделение ответственности
2. **Stateless** — сервер не хранит состояние клиента между запросами
3. **Кешируемость** — ответы могут кешироваться
4. **Единообразие интерфейса** — стандартные методы и форматы
5. **Слоистая архитектура** — клиент не знает, что происходит за его соседом (прокси, балансировщики)
6. **Код по требованию** (опционально) — сервер может передавать код

---

## 5. HTTP-методы в REST

| Метод | Действие | Идемпотентность | Безопасность | Тело запроса |
|-------|----------|-----------------|--------------|--------------|
| **GET** | Получение данных | ✅ Да | ✅ Да | ❌ Нет |
| **POST** | Создание / сложная операция | ❌ Нет | ❌ Нет | ✅ Да |
| **PUT** | Полная перезапись | ✅ Да | ❌ Нет | ✅ Да |
| **PATCH** | Частичное изменение | ✅ Да | ❌ Нет | ✅ Да |
| **DELETE** | Удаление | ✅ Да | ❌ Нет | ❌ Нет |

---

### Идемпотентность
- **Идемпотентный метод** — многократное выполнение даёт тот же результат, что и однократное.
- GET, PUT, DELETE, PATCH — идемпотентны.
- POST — **не идемпотентен** (повторный вызов создаст дубликат).

### Безопасность
- **Безопасный метод** — не изменяет состояние сервера.
- Безопасен только **GET**.
- POST, PUT, PATCH, DELETE — изменяют данные.

## 6. URL — структура и правила

**URL (Uniform Resource Locator)** — адрес ресурса, по которому система понимает, куда отправить запрос.

### Структура URL:
```
https://api.example.com/api/v1/profiles/123/friends?page=2&size=10
```

| Часть | Пример | Назначение |
|-------|--------|------------|
| **Версия** | `/api/v1/` | Указывает версию API |
| **Ресурс (существительное)** | `/profiles/` | Сущность, с которой работаем |
| **Идентификатор** | `/123/` | Конкретный экземпляр (ID, alias) |
| **Вложенный ресурс** | `/friends/` | Связанная сущность |
| **Query-параметры** | `?page=2&size=10` | Фильтрация, пагинация |

---

### Правила построения URL:

| Правило | Хорошо | Плохо |
|---------|--------|-------|
| **Используйте существительные, не глаголы** | `GET /profiles` | `GET /getProfile` |
| **Используйте множественное число** | `/profiles` | `/profile` |
| **Идентификаторы в пути** | `/profiles/123` | `/profiles?id=123` |
| **От большего к меньшему** | `/countries/russia/moscow` | `/moscow/russia` |
| **Не кладите действия в URL** | `POST /profiles/123/friends` | `POST /addFriendToProfile` |

---

## 7. Query-параметры

**Query-параметры** — это `?key=value&key2=value2` в конце URL.

**Используются для:**
- Пагинации (`?page=2&size=10`)
- Фильтрации (`?city=moscow`)
- Сортировки (`?sort=createdAt,desc`)

**Пример:**
```
GET /profiles?city=moscow&isActive=true
```

**Важно:**
- Описывайте типы параметров в документации (строка, число, булево, список)
- Query-параметры — **не для изменения данных**, только для фильтрации и поиска

---

## 8. Тело запроса (Body)

**Body** — используется в POST, PUT, PATCH для передачи данных.

### Формат JSON:
```json
{
  "name": "Аналитик",
  "salary": 150000,
  "skills": [
    {"name": "SQL", "level": 5},
    {"name": "Python", "level": 4}
  ],
  "isActive": true
}
```

### Описания в API должны включать:
- Тип поля (строка, число, булево, объект, массив)
- Обязательность (required / optional)
- Ограничения (длина строки, диапазон чисел, список допустимых значений)

---

## 9. HTTP-статус-коды

### Успешные (2xx)
| Код | Значение | Когда использовать |
|-----|----------|-------------------|
| **200 OK** | Всё хорошо, есть тело | GET, PUT, PATCH |
| **201 Created** | Ресурс создан | POST (создание) |
| **204 No Content** | Всё хорошо, тела нет | DELETE, POST (без возврата) |

### Ошибки клиента (4xx)
| Код | Значение | Когда использовать |
|-----|----------|-------------------|
| **400 Bad Request** | Неверный запрос | Невалидные данные, пропущены обязательные поля |
| **401 Unauthorized** | Не авторизован | Нет токена |
| **403 Forbidden** | Доступ запрещён | Недостаточно прав |
| **404 Not Found** | Ресурс не найден | ID не существует |
| **409 Conflict** | Конфликт | Найдено 2 записи вместо 1, дубликат |

### Ошибки сервера (5xx)
| Код | Значение | Когда использовать |
|-----|----------|-------------------|
| **500 Internal Server Error** | Внутренняя ошибка сервера | Ошибка БД, непредвиденное исключение |

## 10. Тело ответа

### Для пагинации (GET /profiles?page=1&size=10):
```json
{
  "content": [
    { "id": 1, "name": "Иван" },
    { "id": 2, "name": "Мария" }
  ],
  "totalElements": 42,
  "totalPages": 5,
  "page": 1,
  "size": 10
}
```

### Для ошибок:
```json
{
  "errorCode": "VALIDATION_ERROR",
  "message": "Поле 'name' обязательно для заполнения"
}
```

> **Важно:** внутренний `errorCode` помогает фронту различать типы ошибок на клиенте.

---

## 11. Документирование API

В документации API должно быть указано:

| Что описывать | Пример |
|---------------|--------|
| **Метод** | `POST` |
| **URL** | `/api/v1/applications` |
| **Заголовки** | `Authorization: Bearer {token}` |
| **Тело запроса** | JSON-схема с типами и обязательностью |
| **Успешный ответ** | 201 + JSON-схема ответа |
| **Ошибки** | 400, 404, 409, 500 + описание |

---

## 12. CRUD — базовые операции

**CRUD** (Create, Read, Update, Delete) — стандартный набор операций над сущностью.

| Операция | Метод | URL |
|----------|-------|-----|
| **Создать** | `POST` | `/api/v1/applications` |
| **Получить все** | `GET` | `/api/v1/applications` |
| **Получить один** | `GET` | `/api/v1/applications/{id}` |
| **Обновить полностью** | `PUT` | `/api/v1/applications/{id}` |
| **Обновить частично** | `PATCH` | `/api/v1/applications/{id}` |
| **Удалить** | `DELETE` | `/api/v1/applications/{id}` |

---

## 13. Итоговые тезисы (шпаргалка)

1. **API** — контракт между системами («дай мне это, я отвечу так»)
2. **REST** — архитектурный стиль, а не протокол
3. **HTTP-методы**: GET (чтение), POST (создание), PUT (перезапись), PATCH (частичное изменение), DELETE (удаление)
4. **Stateless** — сервер не запоминает состояние клиента между запросами
5. **URL** = существительные, множественное число, версия, идентификаторы в пути
6. **Query-параметры** — для пагинации и фильтрации (`?page=2&size=10`)
7. **Body** — для POST, PUT, PATCH (JSON)
8. **200** = успех, **400** = ошибка клиента, **500** = ошибка сервера
9. **Документируйте всё**: типы, обязательность, ограничения, коды ответов
10. **POST** — не идемпотентен, остальные — идемпотентны
11. **GET** — только для чтения данных, не для изменений
