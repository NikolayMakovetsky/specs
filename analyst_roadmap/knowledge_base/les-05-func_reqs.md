# Конспект лекции №5: Пользовательские и функциональные требования

---

## 1. Что такое пользовательское требование?

**Пользовательское требование** — это задача (действие), которое определённый класс пользователей должен иметь возможность выполнить в системе.

### Три ключевых момента в определении:
1. **Действие** (вместо слова "задача") — что именно пользователь делает
2. **Класс пользователей** — группа пользователей, объединённых по общим признакам
3. **В системе** — ограничение, что действие выполняется именно в рамках проектируемой системы

---

## 2. Три способа описания пользовательских требований

| Способ | Формат | Пример |
|--------|--------|--------|
| **Текстовое описание** | Роль + возможность/ограничение + объект + условия | «Пользователь должен иметь возможность просматривать список товаров без авторизации» |
| **Варианты использования (Use Case)** | Глагол + объект | «Оформить бронирование», «Управлять каталогом» |
| **Пользовательские истории (User Story)** | «Как <роль>, я хочу <цель>, чтобы <причина>» | «Как гость, я хочу зарегистрироваться в отеле, чтобы не тратить время на стойке регистрации» |

---

### 2.1 Текстовое описание
**Формат возможности:**
> Роль + должен иметь возможность + действие + объект + при каких условиях

**Формат ограничения:**
> Роль + должно быть запрещено + действие + объект + при каких условиях

**Примеры:**
- Возможность: «Пользователь должен иметь возможность просматривать список товаров без необходимости авторизации»
- Ограничение: «Пользователю должно быть запрещено удаление сообщений, отправленных другими пользователями»

---

### 2.2 Варианты использования (Use Case)
**Формат:** Глагол + объект

**Примеры:**
- Для отеля: «Зарегистрироваться в отеле», «Управлять доступом в номер»
- Для интернет-магазина: «Купить товар», «Управлять каталогом товаров»

---

### 2.3 Пользовательские истории (User Stories)
**Формат:**
> Как <роль> я хочу <действие/цель> чтобы <выгода/причина>

**Примеры:**
- «Как гость, я хочу зарегистрироваться в отеле, чтобы не тратить время на стойке регистрации»
- «Как менеджер, я хочу управлять каталогом товаров, чтобы на сайте всегда был актуальный перечень товаров»

---

## 3. Диаграмма вариантов использования (Use Case Diagram)

### Основные элементы:
| Элемент | Обозначение | Описание |
|---------|-------------|----------|
| **Актёр (Actor)** | 👤 | Пользователь или внешняя система |
| **Вариант использования** | Овал | Действие в формате «глагол + объект» |
| **Система** | Прямоугольник | Границы системы |
| **Ассоциация** | Линия | Связь актёра с вариантом использования |
| **Включение (Include)** | Стрелка с <<include>> | Обязательный вариант использования внутри другого |
| **Расширение (Extend)** | Стрелка с <<extend>> | Альтернативный (необязательный) сценарий |
| **Обобщение (Generalization)** | Стрелка с пустым треугольником | Наследование (актёр или вариант использования) |

---

### Отношения между вариантами использования:

| Тип | Обозначение | Описание | Обязательность |
|-----|-------------|----------|----------------|
| **Включение (<<include>>)** | Пунктирная стрелка к включаемому варианту | Один вариант всегда включает другой | **Обязательно** |
| **Расширение (<<extend>>)** | Пунктирная стрелка от расширяющего к базовому | Дополнительная возможность | **Необязательно** |
| **Обобщение** | Сплошная стрелка с пустым треугольником | Наследование поведения | - |

---

### Пример диаграммы (интернет-магазин):

**Актёры:** Клиент, Контент-менеджер, Marketplace (внешняя система)

**Варианты использования:**
- Заказать товар (включает: просмотреть каталог)
- Заказать материальный товар (обобщение от "Заказать товар")
- Заказать цифровой товар (обобщение от "Заказать товар")
- Оплатить сразу (расширение для "Заказать товар")
- Управлять каталогом (включает: просмотреть каталог)

**Клиент** может быть:
- Физическое лицо
- Юридическое лицо

## 4. Реестр вариантов использования

**Что это:** Таблица, в которой собраны все варианты использования для разных классов пользователей.

| Класс пользователя | Роль в системе | Варианты использования |
|--------------------|----------------|------------------------|
| Гость | Неавторизованный пользователь | Зарегистрироваться, Авторизоваться, Просмотреть каталог |

## 5. Функциональные требования

**Функциональное требование** — описание требуемого поведения системы в определённых условиях.

### Отличие от пользовательских требований:
| | Пользовательское требование | Функциональное требование |
|---|----------------------------|---------------------------|
| **Точка зрения** | Со стороны пользователя | Со стороны системы |
| **Формулировка** | «Пользователь может...» | «Система должна предоставлять...» |

> **Важно:** В некоторых проектах эти требования не разделяют и описывают вместе.

---

### Форматы функциональных требований:
**Формат возможности:**
> Система должна предоставлять + роль + возможность + действие + объект + при условиях

**Формат ограничения:**
> Система должна запрещать + роль + действие + объект + при условиях

**Примеры:**
- Возможность: «Система должна предоставлять клиенту возможность поиска товаров в каталоге»
- Ограничение: «Система должна запрещать клиенту заказывать товары, которые отсутствуют на складе»

## 6. Шаблон сценария варианта использования

| Поле | Описание | Пример |
|------|----------|--------|
| **Идентификатор** | Уникальный номер | UC-1 |
| **Название** | Краткое наименование | Отправить приглашение на собеседование |
| **Краткое описание** | Что делает вариант | Менеджер отправляет приглашение выбранному кандидату |
| **Действующее лицо (Actor)** | Кто выполняет | Менеджер кадрового агентства |
| **Предусловие** | Что должно произойти до начала | Работодатель согласовал список кандидатов, резюме актуально |
| **Постусловие** | Результат выполнения | Приглашение отправлено кандидату |
| **Цель** | Зачем это нужно | Назначить собеседование с кандидатом |
| **Основной сценарий** | Пошаговое описание (пользователь ↔ система) | 1. Пользователь выбирает список кандидатов... |
| **Альтернативные сценарии** | Исключения и отклонения | Отсутствует канал связи → копирование данных |

---

### Алгоритм написания сценария:
1. **Определить действующих лиц** (актёров)
2. **Составить список вариантов использования**
3. Для каждого варианта:
   - Определить цель и результат
   - Описать основной поток (Happy Path)
   - Дополнить альтернативными сценариями

---

## 7. Пример: сценарий «Отправить приглашение на собеседование»

| Поле | Значение |
|------|----------|
| **Идентификатор** | UC-5 |
| **Название** | Отправить приглашение соискателю на собеседование |
| **Действующее лицо** | Менеджер кадрового агентства |
| **Предусловие** | 1. Работодатель согласовал список кандидатов<br>2. Менеджер согласовал время с работодателем<br>3. Резюме кандидата актуально (активно)<br>4. Пользователь авторизован в системе |
| **Постусловие** | Приглашение отправлено кандидату |
| **Цель** | Назначить собеседование с кандидатом |

### Основной сценарий:
| Шаг | Действие пользователя | Действие системы |
|-----|----------------------|------------------|
| 1 | Переходит к просмотру списка кандидатов | Отображает список кандидатов |
| 2 | Выбирает кандидата | Отображает контактные данные и каналы коммуникации |
| 3 | Выбирает канал связи | - |
| 4 | Формирует текст приглашения | - |
| 5 | Отправляет приглашение | Отправляет приглашение кандидату (email/телеграм) |

### Альтернативный сценарий:
| Шаг | Альтернатива |
|-----|--------------|
| 6.1 | Если автоматизированный канал связи отсутствует, система позволяет скопировать контактные данные пользователем для ручной отправки |

## 8. Связь вариантов использования и функциональных требований

**Один вариант использования** может быть представлен **несколькими функциональными требованиями**.

**Пример:**
- Вариант использования: «Управлять каталогом товаров»
- Функциональные требования:
  1. Система должна предоставлять контент-менеджеру возможность просматривать каталог товаров
  2. Система должна предоставлять контент-менеджеру возможность создавать товары в каталоге
  3. Система должна предоставлять контент-менеджеру возможность редактировать товары
  4. Система должна предоставлять контент-менеджеру возможность удалять товары
  5. Система должна обеспечивать поиск товаров в каталоге

## 9. Практическое задание (Кадровое агентство)

### Шаг 1: Определить классы пользователей и роли

| Класс пользователя | Роль в системе |
|--------------------|----------------|
| Соискатель (Физлицо) | Соискатель |
| Соискатель (ИП) | Соискатель |
| Работодатель | Работодатель |
| Менеджер агентства | Менеджер |
| Руководитель агентства | Руководитель |
| Администратор | Администратор |

---

### Шаг 2: Составить варианты использования

**Для соискателя:**
- Управлять резюме (создание, редактирование, удаление)
- Получать обратную связь по резюме
- Отвечать на приглашения

**Для работодателя:**
- Управлять вакансиями (создание, редактирование, удаление)
- Просматривать список кандидатов
- Согласовывать кандидатов
- Отправлять приглашения

**Для менеджера:**
- Управлять заявками (проверка, корректировка)
- Обрабатывать резюме
- Сопоставлять резюме с вакансиями
- Назначать собеседования
- Формировать отчёты

---

### Шаг 3: Построить диаграмму вариантов использования

1. Нарисовать систему (прямоугольник)
2. Разместить актёров снаружи
3. Разместить варианты использования внутри
4. Соединить актёров с вариантами (ассоциация)
5. Добавить обобщения (например, Соискатель → Физлицо и ИП)
6. Добавить включения (<<include>>) и расширения (<<extend>>)

## 11. Итоговые тезисы (шпаргалка)

1. **Пользовательское требование** = действие пользователя в системе
2. **Функциональное требование** = поведение системы в ответ на действия пользователя
3. Три способа описания: **текст**, **Use Case**, **User Story**
4. **Use Case** = глагол + объект (например, «Управлять заявками»)
5. **User Story** = «Как <роль>, я хочу <цель>, чтобы <выгода>»
6. Включение (<<include>>) — **обязательно**, расширение (<<extend>>) — **необязательно**
7. Для больших систем удобно использовать **реестр вариантов использования**
8. Один вариант использования = несколько функциональных требований
9. **Шаблон сценария:** идентификатор, название, актёр, предусловие, постусловие, цель, основной сценарий, альтернативы
10. При описании сценария всегда указывать **действия пользователя** и **действия системы** парами
