# Конспект лекции №16: Интеграции. Очереди сообщений, WebSocket, SSE, Webhooks

---

## 1. SOAP — протокол интеграции

**SOAP (Simple Object Access Protocol)** — это **строгий протокол** обмена сообщениями.

### Особенности:
- Работает на **XML** (не JSON)
- **Жёсткая спецификация** — всё строго описано в WSDL-файлах
- **Широкая поддержка** корпоративными продуктами
- Используется в **промышленных и отраслевых** проектах

### Недостатки:
- Сложный, громоздкий (описания могут быть на тысячи строк)
- XML парсится тяжелее, чем JSON
- Требует много времени на интеграцию

> **На практике** SOAP редко используется в современных проектах, но встречается в legacy-системах, особенно на производстве и в госсекторе.

## 2. Синхронное vs Асинхронное взаимодействие

### Синхронное взаимодействие (REST, SOAP)
- Клиент отправляет запрос и **ждёт ответа**
- Система блокируется до получения результата
- **Плюсы:** простота, прозрачность, лёгкий дебаг
- **Минусы:** зависимость от доступности системы, узкие места при нагрузке

### Асинхронное взаимодействие (очереди, брокеры)
- Клиент **не ждёт ответа** немедленно
- Отправляет сообщение и продолжает работу
- **Плюсы:** отказоустойчивость, масштабируемость
- **Минусы:** сложность, сложный дебаг

## 3. Брокеры сообщений — общая концепция

**Брокер сообщений** — это посредник между отправителем (продюсером) и получателем (консюмером).

### Основные понятия:
| Термин | Описание |
|--------|----------|
| **Producer** | Отправитель сообщения |
| **Consumer** | Получатель, который читает сообщения |
| **Message** | Данные, передаваемые (чаще всего JSON) |
| **Exchange (RabbitMQ)** | Пространство для маршрутизации |
| **Routing Key** | Ключ, по которому сообщение направляется в нужную очередь |
| **Binding** | Правило: какой routing key → в какую очередь |
| **Queue** | Очередь сообщений для конкретного потребителя |
| **Topic (Kafka)** | Логическое хранилище сообщений, разбитое на партиции |

---

## 4. RabbitMQ (пуш-брокер)

**RabbitMQ** — популярный брокер, работающий по принципу **push**.

### Особенности:
- **Push-модель:** брокер сам отправляет сообщение консюмеру
- Сообщение **удаляется** после подтверждения от консюмера (`ack`)
- **Гибкая маршрутизация** через exchanges, routing keys и bindings
- **Простой** в настройке, лёгкий

### Схема работы:
```
[Producer] → [Exchange] → (Routing Key) → [Queue] → [Consumer]
```

**Пример:**
1. Сообщение отправляется в exchange `notification`
2. У exchange есть bindings: `sms` → очередь SMS-адаптера, `email` → очередь Email-адаптера
3. Брокер копирует сообщение в нужные очереди
4. Консюмеры вычитывают и подтверждают получение

> **Важно:** RabbitMQ удаляет сообщение **только после того, как все консюмеры подтвердили получение**.

## 5. Kafka (пул-брокер)

**Apache Kafka** — распределённый брокер, работающий по принципу **pull**.

### Особенности:
- **Pull-модель:** консюмер сам ходит за сообщениями
- **Сообщения НЕ удаляются** — хранятся по настроенной политике (например, 2 недели)
- **Партиции** — физическое разделение для параллельной обработки
- **Гарантия порядка:** в пределах одной партиции сообщения читаются последовательно
- **Consumer Groups** — группы консюмеров, где каждый читает свою партицию

### Ключевые понятия Kafka:
| Термин | Описание |
|--------|----------|
| **Topic** | Логическое имя для сообщений |
| **Partition** | Физическое разделение внутри топика |
| **Consumer Group** | Группа консюмеров, которые распределяют партиции |
| **Offset** | Смещение, хранимое консюмером (сколько прочитано) |
| **Key** | Ключ, определяющий партицию (один ключ → одна партиция) |

### Важность ключей:
- Чтобы сохранить **порядок сообщений** для одного пользователя, используйте **один и тот же ключ** (например, `userId`)
- Сообщения с одним ключом всегда попадают в одну партицию

## 6. RabbitMQ vs Kafka — сравнение

| Критерий | RabbitMQ | Kafka |
|----------|----------|-------|
| **Модель** | Push (брокер отправляет) | Pull (консюмер забирает) |
| **Удаление сообщений** | Удаляются после ack | Не удаляются, хранятся по политике |
| **Маршрутизация** | Гибкая (exchanges + routing keys) | Нет — только топик + ключ |
| **Порядок сообщений** | Не гарантирован | Гарантирован в партиции |
| **Сложность** | Проще | Сложнее |
| **Нагрузка** | Для средних нагрузок | Для высоких нагрузок |

## 7. WebSocket — двусторонняя связь

**WebSocket** — протокол для **двустороннего** обмена данными через одно постоянное соединение.

### Особенности:
- **Двусторонняя связь:** и клиент, и сервер могут отправлять сообщения
- Одно долгоживущее **TCP-соединение**
- Используется для **чатов, игр, биржевых котировок**
- Внутри передаются обычные HTTP-запросы (POST, GET, JSON)

### Как работает:
1. Клиент и сервер "пожимают руки" (handshake)
2. Устанавливается постоянный канал
3. Сообщения летают туда-обратно без переподключения
4. Периодически отправляются ping/pong для проверки жизни

---

## 8. SSE (Server-Sent Events) — односторонняя связь

**SSE** — технология для отправки событий от сервера к клиенту **в реальном времени**.

### Особенности:
- **Односторонняя связь:** только сервер → клиент
- Долгоживущее соединение
- Используется для **уведомлений, ленты новостей**
- Сервер отправляет события сразу при их возникновении

### Отличие от WebSocket:
| WebSocket | SSE |
|-----------|-----|
| Двусторонняя | Односторонняя (сервер→клиент) |
| Сложнее | Проще |
| Чаты, игры | Уведомления, лента |

---

## 9. Webhooks — триггер по URL

**Webhook** — это механизм, при котором сервер вызывает **заранее переданный URL** при наступлении события.

### Как работает:
1. Клиент сообщает серверу: "Мой URL — `/webhook`"
2. При наступлении события сервер отправляет **POST-запрос** на этот URL
3. Клиент обрабатывает событие

### Примеры:
- Уведомление об оплате
- Триггер после голосования (обновление результатов)
- Интеграции с внешними системами

> **Отличие от SSE:** в Webhooks нет постоянного соединения — это обычный HTTP-запрос.

---

## 10. Итоговые тезисы (шпаргалка)

1. **RabbitMQ** — push-брокер, просто настраивается, удаляет сообщения после ack
2. **Kafka** — pull-брокер, не удаляет сообщения, гарантирует порядок через партиции
3. **WebSocket** — двусторонняя связь, постоянное соединение (чаты, игры)
4. **SSE** — односторонняя связь, сервер → клиент (уведомления)
5. **Webhooks** — сервер вызывает URL клиента при событии (без постоянного соединения)
6. Асинхронная связь **отказоустойчивее**, но сложнее в дебаге
7. В RabbitMQ маршрутизация через **exchanges + routing keys + bindings**
8. В Kafka маршрутизация через **топики + ключи + партиции**
