# Конспект лекции №11: Введение в архитектуру ПО

---

## 1. Переход к технической части

Курс делится на две большие части:
- **Бизнес-аналитика**: бизнес-процессы, юзер-стори, функциональные требования, сценарии
- **Техническая часть (архитектура)**: протоколы, авторизация, микросервисы, интеграции, диаграммы последовательностей

---

## 2. Зачем системному аналитику знать архитектуру?

1. **Работа в существующем архитектурном ландшафте** — большинство проектов уже имеют готовую архитектуру
2. **Понимание технических ограничений** — как интегрировать новые требования
3. **Общение с командой на одном языке** — с архитектором, тимлидом, разработчиками
4. **Замена архитектора** — на небольших проектах аналитик может взять часть функций архитектора

---

## 3. Что такое архитектура системы?

**Архитектура системы** — это:
- Принципиальная организация системы
- Совокупность её элементов
- Взаимодействие между элементами
- Принципы развития и проектирования системы

> В каждой команде может быть свой архитектурный подход.

---

## 4. Трёхслойная архитектура (3-Tier Architecture)

Это классическая архитектура, разделяющая систему на три логических слоя:

### 4.1 Слой представления (Presentation Layer)
- **Что это**: интерфейс пользователя (UI)
- **Где находится**: на клиентской стороне (фронтенд)
- **Пример**: мобильное приложение, веб-сайт

### 4.2 Слой бизнес-логики (Business Logic Layer / Backend)
- **Что это**: обработка команд, бизнес-правила, логика приложения
- **Где находится**: на серверной стороне
- **Пример**: обработка запросов, авторизация, валидация данных

### 4.3 Слой доступа к данным (Data Access Layer / Database)
- **Что это**: хранение и управление данными
- **Где находится**: база данных (БД)
- **Пример**: PostgreSQL, MySQL, Redis

---

### Визуализация трёхслойной архитектуры:

```
[Клиент (фронтенд)] ←→ [Сервер (бэкенд)] ←→ [База данных (БД)]
```

---

## 5. Фронтенд vs Бэкенд

| Компонент | Описание | Технологии |
|-----------|----------|------------|
| **Фронтенд** | То, что видит пользователь (UI) | HTML, CSS, JavaScript, React, Swift, Kotlin |
| **Бэкенд** | Серверная логика, обработка данных | Java, Python, C#, Node.js, Go, PHP |
| **БД** | Хранение структурированных данных | PostgreSQL, MySQL, Oracle, MongoDB |

---

## 6. Архитектурные характеристики (Quality Attributes)

При проектировании архитектуры нужно учитывать следующие свойства:

| Характеристика | Описание |
|----------------|----------|
| **Доступность** | Система работает 24/7, доступна в любой момент (цель ~99.9%) |
| **Надёжность** | Вероятность непрерывной работы в течение времени |
| **Тестируемость** | Насколько легко тестировать отдельные модули и систему в целом |
| **Масштабируемость** | Способность увеличивать мощность при росте нагрузки |
| **Безопасность** | Защита от атак, утечек данных |
| **Гибкость** | Лёгкость внедрения новых требований |
| **Отказоустойчивость** | Способность восстанавливаться после сбоев |
| **Эластичность** | Способность реагировать на нестандартную нагрузку |
| **Восстанавливаемость** | Восстановление после отказов |
| **Производительность** | Время отклика, количество запросов в секунду |
| **Развёртываемость** | Лёгкость развёртывания на разных окружениях |
| **Обучаемость** | Способность системы адаптироваться (ML, AI) |

---

### Два типа масштабирования:

| Тип | Описание | Применение |
|-----|----------|------------|
| **Вертикальное** | Улучшение железа (CPU, RAM, диск) | Реляционные БД |
| **Горизонтальное** | Добавление новых серверов / экземпляров | Микросервисы, NoSQL |

---

## 7. Практический пример: трёхслойная архитектура для кадрового агентства

### Сценарий: работодатель создаёт заявку на подбор персонала

**1. Авторизация на фронтенде**
- Клиент вводит логин/пароль
- Запрос на бэкенд → проверка в БД → ответ о статусе

**2. Получение списка заявок (GET /applications)**
- Клиент запрашивает свои заявки
- Бэкенд обращается к БД
- БД возвращает данные → бэкенд формирует ответ → фронтенд отображает

**3. Создание новой заявки (POST /applications)**
- Клиент заполняет форму на фронтенде
- Запрос с данными на бэкенд
- Валидация данных → запись в БД → ответ об успехе/ошибке

**4. Обработка заявки сотрудником кадрового агентства**
- Сотрудник авторизуется в админке (отдельный фронтенд)
- Видит новые заявки (GET /applications?status=new)
- Меняет статус заявки (PUT /applications/{id}/status)
- Клиент видит обновлённый статус

---

### Схема взаимодействия:

```
[Фронтенд (клиент)] → GET /applications → [Бэкенд] → SQL-запрос → [БД]
[Фронтенд (клиент)] ← список заявок ← [Бэкенд] ← данные ← [БД]

[Фронтенд (админка)] → PUT /applications/123/status → [Бэкенд] → UPDATE → [БД]
[Фронтенд (админка)] ← статус обновлён ← [Бэкенд] ← OK ← [БД]
```

---

## 8. Итоговые тезисы (шпаргалка)

1. **Архитектура** — это организация системы, её элементы и их взаимодействие
2. **Трёхслойная архитектура**: представление → бизнес-логика → данные
3. **Фронтенд** — то, что видит пользователь
4. **Бэкенд** — серверная часть, обрабатывающая логику
5. **БД** — хранение и управление данными
6. Архитектурные характеристики: **доступность, производительность, масштабируемость, безопасность** и др.
7. В разных проектах приоритеты характеристик могут отличаться
8. Аналитику нужно понимать архитектуру, чтобы:
   - Общаться с разработчиками на одном языке
   - Оценивать реализуемость требований
   - Проектировать интеграции
