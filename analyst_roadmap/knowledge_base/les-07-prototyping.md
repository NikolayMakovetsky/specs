# Конспект лекции №7: Проектирование пользовательского интерфейса (UI) и прототипирование

---

## 1. Что такое UI (User Interface)?

**UI (Пользовательский интерфейс)** — это визуальная составляющая системы:
- Как выглядят экранные формы
- Цветовая гамма
- Композиция
- Оформление графических элементов

### Виды UI в зависимости от цели:
| Тип продукта | Особенности UI |
|--------------|----------------|
| **Прикладные B2B-продукты** | Минималистичный, упрощённый, функциональный |
| **Массовые продукты (соцсети, маркетплейсы)** | Красочный, проработанный, с «плюшками» |

---

## 2. Основные принципы проектирования UI

### 2.1 Сигнализировать состояние системы
Пользователь должен получать обратную связь на свои действия.

**Примеры:**
- Индикатор загрузки («Идёт загрузка...»)
- Прогресс-бар при проигрывании музыки
- Статус заказа на маркетплейсе

---

### 2.2 Использовать опыт и привычки пользователя
Применяйте знакомые пользователю паттерны и ментальные модели.

**Примеры:**
- «Бургер-меню» (три полоски) для навигации
- Корзина для добавления товаров
- Поиск в виде лупы

---

### 2.3 Экономить усилия пользователя
Система должна запоминать данные, чтобы не заставлять пользователя вводить их повторно.

**Примеры:**
- Генерация сложного пароля
- Запоминание последнего пункта выдачи заказа
- Автодополнение адресов

---

### 2.4 Продумывать пути выхода (обратная связь перед критическими действиями)

**Примеры:**
- Подтверждение перед удалением
- Автосохранение черновиков (как на Госуслугах)
- Восстановление файлов после аварийного завершения (Excel)

---

### 2.5 Возможность настройки под пользователя

**Примеры:**
- Настройка горячих клавиш
- Изменение размера шрифта
- Версия для слабовидящих
- Пропуск онбординга (вводного курса)

---

### 2.6 Консистентность (единообразие)

| Тип | Описание |
|-----|----------|
| **Визуальная** | Единый стиль во всей системе |
| **Функциональная** | Одинаковые элементы ведут себя одинаково |

**Примеры:**
- Название раздела («Личный кабинет») не меняется на других страницах
- Навигационное меню всегда на одном месте
- Кнопки для похожих действий выглядят одинаково

---

### 2.7 Предотвращение ошибок (недвусмысленность)

**Примеры:**
- Иконка «крестик» — для отмены, «корзина» — для удаления
- Достаточное расстояние между кнопками
- Валидация полей с подсветкой ошибок

---

### 2.8 Дозирование информации
Не вываливайте на пользователя всё сразу — выдавайте частями.

**Пример:**
- Пошаговое заполнение заявки на Госуслугах
- Оформление полиса ОСАГО поэтапно

---

## 3. Четыре этапа проектирования UI

| Этап | Описание | Исполнитель |
|------|----------|-------------|
| **1. Исследование** | Сбор требований, подготовка спецификации | Аналитик |
| **2. Прототипирование** | Создание «скелета» системы (wireframe) без декора | Аналитик (или совместно с дизайнером) |
| **3. Дизайн (опционально)** | Наложение цветов, шрифтов, финальной эстетики | UI/UX-дизайнер |
| **4. Разработка, тестирование, запуск** | Программирование, проверка и ввод в эксплуатацию | Команда разработки |

---

## 4. Прототип vs Дизайн

| | Прототип (Wireframe) | Дизайн (Mockup) |
|---|----------------------|-----------------|
| **Цель** | Показать структуру и взаимодействие | Показать финальный внешний вид |
| **Детали** | Схема, «скелет», чёрно-белое | Цвета, шрифты, иконки, графика |
| **Инструменты** | Balsamiq, Figma (в режиме прототипа) | Figma, Adobe XD, Sketch |
| **Кто делает** | Аналитик | Дизайнер |

---

## 5. Инструменты для прототипирования

| Инструмент | Описание |
|------------|----------|
| **Figma** | Самый популярный, позволяет и прототипы, и дизайн |
| **Balsamiq** | Быстрое создание «набросков» (скелетов) |
| **Adobe XD** | Полнофункциональный инструмент |
| **Axure RP** | Для сложных интерактивных прототипов |

## 6. Экранные формы (Screen Forms)

### Что это?
Таблица, которая описывает элементы пользовательского интерфейса и их поведение.

### Структура экранной формы:

| Столбец | Описание |
|---------|----------|
| **№** | Номер элемента |
| **Элемент прототипа** | Название элемента (поле, кнопка, список) |
| **API-запрос / действие** | Какой запрос вызывается при взаимодействии |
| **Комментарий** | Дополнительная информация (JSON, ограничения) |

### Зачем нужны экранные формы?
- Связь между сценариями (Use Cases) и интерфейсом
- Помощь frontend-разработчикам
- Снижение количества вопросов к аналитику
- Единый источник истины для команды

> **Важно:** Экранные формы могут отсутствовать в сплочённых командах, где все понимают друг друга с полуслова.

## 7. Практический пример

### Сценарий: Просмотр списка заявок на подбор персонала

**Элементы интерфейса:**

| № | Элемент | API-запрос | Комментарий |
|---|---------|------------|-------------|
| 1 | Поле поиска | GET /requests?query=... | Поиск по ключевому слову |
| 2 | Фильтр: дата создания | — | DatePicker |
| 3 | Фильтр: заказчик | POST /requests/filter | Тело запроса в JSON (динамический список) |
| 4 | Фильтр: позиция | POST /requests/filter | Справочник должностей |
| 5 | Фильтр: статус заявки | POST /requests/filter | Выпадающий список (ComboBox) |
| 6 | Кнопка «Применить» | POST /requests/filter | Отправка фильтров на сервер |
| 7 | Таблица (Grid) с заявками | GET /requests | Отображение списка с колонками: №, заказчик, должность, статус |

---

### Прототип страницы (в Balsamiq):

```
[Поле поиска] (строка ввода)

[Фильтр: Дата] [Фильтр: Заказчик ▼] [Фильтр: Позиция ▼] [Фильтр: Статус ▼] [Применить]

| № | Заказчик | Должность | Статус |
|---|----------|-----------|--------|
| 1 | ООО "Ромашка" | Java-разработчик | В работе |
| 2 | ИП "Весна" | Тестировщик | Закрыта |
| 3 | АО "Прогресс" | Аналитик | На согласовании |
```

## 8. Взаимодействие аналитика с дизайнером и разработчиками

### Ключевые моменты:
1. **Прототип сначала утверждается с заказчиком/владельцем продукта** — это тоже требования
2. **Дизайнер подключается после утверждения прототипа** — он дорабатывает визуал
3. **Аналитик и дизайнер работают совместно** — дизайнер уточняет поведение элементов
4. **Сложные элементы согласовываются с frontend-разработчиками** — проверка реализуемости
5. **Финальный дизайн тоже утверждается заказчиком**
6. **Макеты линкуются с задачами разработчиков** в системе управления (Jira, Confluence)

---

## 9. Итоговые тезисы (шпаргалка)

1. **UI** — это внешний вид системы, а не её логика
2. Основные принципы UI: **обратная связь, привычки пользователя, экономия усилий, выходы из сценариев, настройка, консистентность, предотвращение ошибок, дозирование информации**
3. Проектирование UI проходит **4 этапа**: исследование → прототипирование → дизайн → разработка
4. **Прототип** — это «скелет» системы, без цветов и декора
5. **Дизайн** — финальная эстетика (цвета, шрифты, иконки)
6. **Экранные формы** — таблица с описанием элементов UI и их поведения
7. Все прототипы и дизайн **согласовываются с заказчиком**
8. Аналитик, дизайнер и разработчики работают **совместно** на всех этапах
9. Хороший прототип + экранные формы = **меньше вопросов от команды**
