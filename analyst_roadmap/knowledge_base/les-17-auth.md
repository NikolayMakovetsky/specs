# Конспект лекции №17: Идентификация, Аутентификация, Авторизация

---

## 1. Три ключевых понятия

| Понятие | Описание | Пример |
|---------|----------|--------|
| **Идентификация** | Определение, **кто** пользователь | Ввод логина, email, номера телефона |
| **Аутентификация** | Проверка подлинности — **действительно ли это тот пользователь** | Ввод пароля, SMS-кода, отпечатка пальца |
| **Авторизация** | Определение, **что можно делать** после входа | Доступ к админке, просмотр личных данных |

> **Важно:** Идентификация и аутентификация часто происходят вместе (ввод логина и пароля), но на сервере это два разных процесса.

---

### Процесс на сервере:
1. Пользователь вводит логин → сервер **идентифицирует** его (находит в БД)
2. Пользователь вводит пароль → сервер **аутентифицирует** (проверяет пароль)
3. После успеха сервер выдаёт **токен** → пользователь теперь **авторизован**
4. При каждом запросе сервер проверяет токен и права → **авторизация**

---

## 2. Открытые vs Закрытые методы API

### Открытые (публичные) методы
- Не требуют авторизации
- Пример: просмотр каталога товаров, форма обратной связи

### Закрытые (приватные) методы
- Требуют передачи **креденшелов** (токена)
- Пример: просмотр личного кабинета, оформление заказа

---

## 3. HTTP-коды при авторизации

| Код | Значение | Причина |
|-----|----------|---------|
| **401 Unauthorized** | Аутентификация не пройдена | Нет токена или он невалидный |
| **403 Forbidden** | Доступ запрещён | Токен есть, но недостаточно прав (не та роль) |

> **Как запомнить:** 401 — проблема с **токеном**, 403 — проблема с **ролями**.

---

## 4. Способы передачи креденшелов

### 4.1 Basic Authentication

- Логин и пароль объединяются в строку `login:password`
- Кодируются в **Base64**
- Передаются в заголовке: `Authorization: Basic base64string`

**Пример:**
```
Authorization: Basic bG9naW46cGFzc3dvcmQ=
```

**Недостатки:**
- Base64 — это **кодирование, а не шифрование** (легко расшифровать)
- Креденшелы передаются с **каждым запросом**
- Используется только в простых или внутренних системах

---

### 4.2 Bearer Token (JWT)

**JWT (JSON Web Token)** — стандартный формат токена, который содержит информацию о пользователе и правах.

**Структура JWT:**
```
xxxxx.yyyyy.zzzzz
```

| Часть | Название | Что содержит |
|-------|----------|--------------|
| **1-я** | Header | Тип токена (JWT), алгоритм шифрования |
| **2-я** | Payload | Данные о пользователе (логин, роль, время жизни) |
| **3-я** | Signature | Подпись, защищающая от подделки |

**Передача в заголовке:**
```
Authorization: Bearer <JWT-токен>
```

---

### Как формируется подпись (Signature):
```
Signature = SHA256(
    base64UrlEncode(header) + "." + base64UrlEncode(payload),
    secret_key
)
```

- Сервер знает **секретный ключ** (`secret_key`)
- Если кто-то изменит payload (например, роль с user → admin), подпись станет невалидной
- Без секретного ключа подпись нельзя подделать

---

## 5. Access Token vs Refresh Token

| Тип | Время жизни | Назначение |
|-----|-------------|------------|
| **Access Token** | Короткое (5–15 минут) | Передаётся в каждом запросе для авторизации |
| **Refresh Token** | Долгое (дни, месяцы) | Используется для получения нового Access Token |

### Как работает связка:

1. Пользователь вводит логин+пароль
2. Сервер выдаёт **Access Token** и **Refresh Token**
3. Frontend хранит оба токена
4. При каждом запросе передаётся **Access Token**
5. Когда Access Token истекает → приходит `401`
6. Frontend отправляет **Refresh Token** на специальный endpoint
7. Сервер выдаёт **новый Access Token**
8. Пользователь продолжает работу без повторного ввода пароля

**Зачем это нужно?**
- Access Token живёт мало — его сложнее украсть
- Если роль пользователя изменилась (стал админом), новый Access Token получит актуальные права
- Refresh Token можно отозвать отдельно, если он скомпрометирован

## 6. Практический пример: проектирование API для авторизации

### 1. Авторизация (логин + пароль)

**Запрос:**
```
POST /api/v1/auth
Content-Type: application/json

{
  "login": "user@example.com",
  "password": "qwerty123"
}
```

**Ответ (успех):**
```
200 OK
{
  "accessToken": "eyJhbGciOiJIUzI1NiIs...",
  "refreshToken": "eyJhbGciOiJIUzI1NiIs..."
}
```

**Ответ (ошибка):**
- `404 Not Found` — пользователь не найден (идентификация не пройдена)
- `401 Unauthorized` — пароль неверный (аутентификация не пройдена)

---

### 2. Получение данных (закрытый метод)

**Запрос:**
```
GET /api/v1/applications
Authorization: Bearer <accessToken>
```

**Ответ:**
- `200 OK` — данные выданы
- `401 Unauthorized` — Access Token истёк или невалиден
- `403 Forbidden` — недостаточно прав

---

### 3. Обновление Access Token

**Запрос:**
```
POST /api/v1/auth/refresh
Authorization: Bearer <refreshToken>
```

**Ответ:**
```
200 OK
{
  "accessToken": "new_access_token..."
}
```

## 7. Дополнительные понятия

### Single Sign-On (SSO) / OAuth
- Пользователь авторизуется через внешний сервис (Google, Яндекс, VK)
- Сервис-провайдер выдаёт токен
- Сайт принимает этот токен и доверяет ему

### Keycloak
- Отдельный сервис для управления аутентификацией и авторизацией
- Хранит пользователей, роли, выдаёт токены
- Используется в крупных корпоративных проектах

### Хранение токенов
- Токены хранятся в **cookies** или **localStorage** на фронтенде
- Refresh Token должен храниться безопасно (HTTP-only cookie)

---

## 8. Итоговые тезисы (шпаргалка)

1. **Идентификация** — понять, **кто** пользователь (логин/почта)
2. **Аутентификация** — подтвердить, что это **он** (пароль, код)
3. **Авторизация** — определить, что **можно делать** (роли, права)
4. **Basic Auth** — логин+пароль в Base64 (слабая защита)
5. **Bearer Token (JWT)** — стандартный токен, состоит из Header + Payload + Signature
6. **Access Token** — короткоживущий, передаётся в каждом запросе
7. **Refresh Token** — долгоживущий, нужен для обновления Access Token
8. **401** — проблема с токеном, **403** — проблема с правами
9. JWT хранит **роль** в Payload, что позволяет не ходить в БД за каждым запросом
10. При смене роли пользователь получает новый Access Token с обновлённой ролью
