# Конспект лекции №8: Базы данных (БД) и СУБД

---

## 1. Данные и их типы

**Данные** — это информация, представленная в форме, пригодной для обработки с помощью компьютера.

### Основные типы данных:
- Числовые
- Текстовые
- Графические
- Аудио/видео

---

## 2. Базы данных и СУБД

**База данных (БД)** — набор информации, хранящейся в электронном виде.

**Система управления базами данных (СУБД)** — программное обеспечение для взаимодействия пользователя с данными (создание, чтение, обновление, удаление).

> **Важно:** Когда говорят «база данных», чаще всего подразумевают именно СУБД.

---

## 3. Типы баз данных

### 3.1 Реляционные (SQL) базы данных
Данные хранятся в виде таблиц (столбцы → атрибуты, строки → записи). Сущности связаны между собой отношениями.

**Популярные реляционные СУБД:**

| СУБД | Особенности |
|------|-------------|
| **PostgreSQL** | Высокая функциональность, производительность, open-source, популярна в коммерческих проектах |
| **MySQL** | Работает с таблицами разных форматов, до 50 млн записей, интегрируется с другими СУБД |
| **Microsoft SQL Server** | Автоматизация задач, управление памятью, совместимость с Excel и Access |
| **SQLite** | Компактная, не использует сервер, данные на одном устройстве, подходит для небольших проектов |
| **Oracle** | Кроссплатформенная, без ограничений по объёму, подходит для крупных проектов |

> **Язык запросов:** SQL (Structured Query Language)

---

### 3.2 Нереляционные (NoSQL) базы данных
Данные хранятся в другом виде, отличном от таблиц. Появились из-за потребности в хранении неструктурированных данных.

#### Типы NoSQL-баз данных:

| Тип | Описание | Пример СУБД |
|-----|----------|-------------|
| **Ключ-значение (Key-Value)** | Данные как словарь {ключ: значение}, очень быстрые запросы | **Redis** (в памяти) |
| **Документо-ориентированные** | Хранение иерархических структур (JSON, XML) | **MongoDB** |
| **Поисковые** | Поиск по большим неструктурированным данным (логи) | **Elasticsearch**, **OpenSearch** |
| **Колоночные** | Хранение по столбцам, для аналитики больших данных | **Cassandra**, **ClickHouse** |
| **Графовые** | Моделирование сложных отношений (узлы + рёбра) | (примеры не указаны) |

---

## 4. Масштабирование баз данных

| Тип | Описание | Характерно для |
|-----|----------|----------------|
| **Вертикальное** | Улучшение железа (память, процессор) на одном сервере | Реляционных БД |
| **Горизонтальное** | Разделение данных между разными серверами | Нереляционных БД |

---

## 5. Нормализация и денормализация

| Процесс | Описание |
|---------|----------|
| **Нормализация** | Структурирование данных для минимизации избыточности и зависимостей |
| **Денормализация** | Обратный процесс — отказ от нормальных форм для упрощения поиска данных |

---

## 6. Свойства ACID (для реляционных БД)

| Свойство | Описание |
|----------|----------|
| **Атомарность (Atomicity)** | Транзакция выполняется целиком или не выполняется вообще |
| **Согласованность (Consistency)** | Записываются только достоверные данные, соответствующие правилам |
| **Изолированность (Isolation)** | Незавершённые транзакции изолированы друг от друга |
| **Долговечность (Durability)** | Данные сохраняются даже при сбоях системы |

---

## 7. Плюсы и минусы реляционных БД

### Плюсы:
- Хорошо подходят для структурированных данных
- Высокий уровень безопасности (разграничение доступа)
- Защита от потери данных (ACID)

### Минусы:
- Недостаток гибкости — сложно работать с неструктурированными данными
- Сложность передачи информации между решениями
- Только вертикальное масштабирование (работа на одном сервере)

---

## 8. Плюсы и минусы нереляционных БД

### Плюсы:
- Хранение неструктурированной информации
- Облачные вычисления и горизонтальное масштабирование
- Быстрая разработка (меньше подготовительных действий)

### Минусы:
- Привязка к конкретной СУБД (нет универсального языка)
- Ограниченная ёмкость встроенного языка запросов
- Сложность быстрого перехода между разными NoSQL-решениями

---

## 9. CAP-теорема

**CAP-теорема** гласит, что в распределённой системе можно обеспечить **не более двух** из трёх свойств:

| Свойство | Описание |
|----------|----------|
| **Consistency (Согласованность)** | Все копии данных одинаковы в любой момент времени |
| **Availability (Доступность)** | Система отвечает на запрос в любое время, даже при сбоях |
| **Partition Tolerance (Устойчивость к распределению)** | Система работает даже при потере связи между компонентами |

> **На практике:** нельзя реализовать все три свойства одновременно, нужно выбирать приоритеты для конкретной системы.

---

## 10. Роль аналитика в работе с БД

| Что делает аналитик | Комментарий |
|---------------------|-------------|
| **Предлагает выбор СУБД** | Может высказать мнение, но итоговое решение — за архитектором |
| **Проектирует модель данных** | Описывает сущности, их атрибуты, типы данных и связи |
| **Формализует требования** | Описывает, какие данные нужно хранить для выполнения функциональных требований |
| **Согласовывает с командой** | Обсуждает модель с архитектором и разработчиками |

---

## 11. Итоговые тезисы (шпаргалка)

1. **Данные** — информация для компьютерной обработки
2. **БД** — набор данных, **СУБД** — инструмент для управления БД
3. **Реляционные (SQL)** — таблицы + связи, строгая структура
4. **Нереляционные (NoSQL)** — гибкость, неструктурированные данные
5. **ACID** — свойства надёжности реляционных БД
6. **CAP-теорема** — выбор двух из трёх свойств в распределённых системах
7. На проектах часто используют **несколько СУБД одновременно** (SQL + Redis + Elasticsearch)
8. Аналитик проектирует **модель данных**, но выбор СУБД — за архитектором
