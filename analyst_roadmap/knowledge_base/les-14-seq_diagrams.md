# Конспект лекции №14: Архитектурные нотации и диаграммы последовательности

---

## 1. Зачем нужны нотации для описания архитектуры?

Архитектуру можно описывать разными способами:
- **Текстом** — «сервис А делает то-то, сервис Б делает то-то»
- **Таблицами** — список сервисов, их функции, параметры
- **Графически** — любые символы (хоть голуби и кошки)

> **Проблема:** без стандартов каждый рисует по-своему, и никто никого не понимает.

**Нотации** — это согласованные правила и символы для описания архитектуры, чтобы все участники команды говорили на одном языке.

---

## 2. Популярные архитектурные нотации

### 2.1 AADL (Architecture Analysis & Design Language)

**Что это:** полноценная система моделирования с текстовым языком и графическими элементами.

**Особенности:**
- Моделирует систему на **трёх уровнях**:
  - Программный (классы, подклассы, взаимодействия)
  - Физический (память, скорость, задержки)
  - Абстрактный (виртуальные машины, контейнеры)
- Позволяет **автоматически рассчитывать нагрузку** на систему
- Анализирует **надёжность, отказоустойчивость**

**Где используется:** авиация, морское судоходство, оборонная промышленность, крупные промышленные системы.

**Минусы:** высокий порог вхождения, сложный язык, мало информации на русском.

---

### 2.2 SysML (Systems Modeling Language)

**Что это:** расширение UML 2.0 для моделирования систем (не только программных).

**Особенности:**
- Работает с **блоками** (физическими или абстрактными)
- Иерархическая структура: блок → части → порты (входы/выходы) → сигналы (потоки данных/энергии)
- Есть комментарии для пояснений

**Пример:** можно описать комнату с выключателем, светом, электричеством.

---

### 2.3 ArchiMate

**Что это:** самая популярная нотация в российском сегменте. Учитывает **бизнес-контекст**.

**Особенности:**
- Работает со **слоями**:
  - **Бизнес-слой** — стратегия, цели, бизнес-процессы
  - **Прикладной слой** — приложения, сервисы, их взаимодействие
  - **Технологический слой** — серверы, сети, инфраструктура
- **Цветовой код** (неофициальный, но распространённый):
  - 🟢 Зелёный — бизнес
  - 🔵 Синий — техническая часть
  - 🟡 Жёлтый — детали реализации

**Плюсы:**
- Можно показать систему с разных точек зрения
- Единый контекст для бизнеса и разработки
- Поэтапное углубление

---

### 2.4 4+1 View Model

**Что это:** первая модель, предложившая рассматривать систему **с 4 разных точек зрения + 1**.

| Уровень | Диаграмма UML | Описание |
|---------|---------------|----------|
| **Логический** | Диаграмма классов / состояний | Сущности, состояния, процессы |
| **Процессорный** | Диаграмма последовательности | Взаимодействие между компонентами |
| **Разработческий** | Диаграмма пакетов / компонентов | Модули, библиотеки, классы |
| **Физический** | Диаграмма развертывания | Серверы, железо, сеть |
| **+1 (Сценарии)** | Use Case диаграмма | Пользовательские сценарии |

## 3. Модель C4 (модель с 4 уровнями)

**C4** — это эволюция 4+1, но без жёсткой привязки к ООП и UML.

**Главная идея:** **иерархическое углубление** (как приближение в карте).

**Уровни:**

| Уровень | Название | Что показывает | Пример |
|---------|----------|----------------|--------|
| **Уровень 1** | Контекст (Context) | Кто и с кем взаимодействует на верхнем уровне | Пользователь ↔ Банковская система ↔ Email-система |
| **Уровень 2** | Контейнеры (Containers) | Из каких частей состоит система | Веб-приложение, мобильное приложение, API, БД |
| **Уровень 3** | Компоненты (Components) | Из чего состоит каждый контейнер | Контроллеры, адаптеры, сервисы |
| **Уровень 4** | Код (Code) | Диаграмма классов (почти не используется аналитиками) | Классы, интерфейсы, методы |

### Как это выглядит:

**Уровень 1 — Контекст**
```
[Пользователь] → [Система] → [Email-система]
```

**Уровень 2 — Контейнеры**
```
[SPA (веб)] → [API] → [БД]
[Моб. приложение] → [API] → [Email-шлюз]
```

**Уровень 3 — Компоненты**
```
[API] содержит: [Контроллер пользователей] → [Адаптер email]
```

---

### C4 vs UML — как соотносятся:

| Уровень C4 | Диаграмма UML |
|------------|---------------|
| Контекст | Use Case (сценарии использования) |
| Контейнеры | Компонентная диаграмма, диаграмма развертывания |
| Компоненты | Компонентная диаграмма, диаграмма пакетов |
| Код | Диаграмма классов |

### C4 + 3 дополнительных диаграммы

Помимо 4 уровней, создатели C4 предложили ещё 3 диаграммы:

1. **Системный ландшафт** — контекстное окружение системы
2. **Динамическая схема** — как элементы взаимодействуют в процессе (Sequence Diagram)
3. **Схема развертывания** — физическое размещение (серверы, контейнеры, IP)

## 4. Диаграмма последовательности (Sequence Diagram)

### Что это?
Диаграмма, которая показывает **взаимодействие объектов во времени** — как запросы и ответы ходят между сервисами.

**Особенности:**
- Каждый участник — **чёрный ящик** (мы видим только вход/выход, а не внутреннюю логику)
- Показывает **порядок вызовов** и **жизненный цикл** объектов
- **Не показывает** внутреннее устройство сервисов

---

### Основные элементы диаграммы:

| Элемент | Обозначение | Описание |
|---------|-------------|----------|
| **Актёр (Actor)** | 👤 | Пользователь, который инициирует действие |
| **Объект / Сервис** | Прямоугольник | Любой участник (сервис, БД, очередь) |
| **Лайфлайн (Lifeline)** | Вертикальная пунктирная линия | Жизненный цикл объекта во времени |
| **Синхронное сообщение** | Закрашенная стрелка → | Запрос с ожиданием ответа |
| **Асинхронное сообщение** | Стрелка → с открытой головкой | Запрос без ожидания ответа |
| **Ответное сообщение** | Пунктирная стрелка ← | Ответ на запрос |
| **Фрейм (Frame)** | Рамка вокруг группы действий | Группировка, альтернатива, цикл |

---

### Виды фреймов:

| Фрейм | Описание |
|-------|----------|
| **alt** | Альтернативные сценарии («если А, то так, если Б, то иначе») |
| **opt** | Опциональный блок (может быть, а может не быть) |
| **loop** | Циклическое повторение действий N-раз |
| **ref** | Ссылка на другую диаграмму |
| **group** | Просто логическая группировка |

---

### Пример диаграммы последовательности

**Сценарий:** соискатель авторизуется, создаёт заявку, система ищет совпадения и отправляет уведомления.

```
[Соискатель] → [Фронтенд] → [Сервис заявок] → [База данных]
              ↓               ↓                  ↓
[Авторизация] [GET /profile] [CREATE /applications] [INSERT]
              ↓               ↓                  ↓
[Ответ]       [200 OK]       [200 OK]           [201]
              ↓                                 ↓
[Мэтчер] ←── [Сервис заявок] → [Сервис уведомлений] → [Email]
              ↓               ↓                  ↓
[Поиск совпадений] [POST /notify] [Отправка письма]
```

На диаграмме:
- Каждая стрелка — это **запрос** (имя метода/команды)
- Для каждого запроса есть **ответ** (пунктирная стрелка обратно)
- Группировка `alt` показывает: если совпадения есть → уведомить; если нет → ничего не делать

## 5. PlantUML — инструмент для диаграмм

**PlantUML** — это инструмент для создания диаграмм **через код**.

### Особенности:
- Открытый исходный код
- Поддерживает почти все типы UML-диаграмм
- Есть текстовый синтаксис, как у языка программирования
- Можно встраивать в Confluence, Jira, IDEA, VS Code
- Есть **онлайн-редактор** (PlantUML Web)
- Документация на русском языке

### Простой пример:
```
@startuml
actor "Соискатель" as user
participant "Фронтенд" as front
participant "Сервис заявок" as app
database "БД" as db

user -> front: Создать заявку
front -> app: POST /applications
app -> db: INSERT
db --> app: OK
app --> front: 201 Created
front --> user: Успех
@enduml
```

## 6. Итоговые тезисы (шпаргалка)

1. **Нотации** — это стандартные правила описания архитектуры, чтобы все понимали друг друга
2. Популярные нотации: **AADL, SysML, ArchiMate, 4+1, C4**
3. **C4** — иерархическая модель: контекст → контейнеры → компоненты → код
4. **Диаграмма последовательности** — показывает взаимодействие объектов во времени
5. В диаграмме последовательности: **запрос** = закрашенная стрелка →, **ответ** = пунктирная стрелка ←
6. Можно показывать: **альтернативы (alt)**, **циклы (loop)**, **опциональные блоки (opt)**
7. **PlantUML** — инструмент для создания диаграмм кодом, популярен у аналитиков
