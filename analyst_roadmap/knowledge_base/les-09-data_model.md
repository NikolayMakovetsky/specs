# Конспект лекции №9: Моделирование данных и ER-диаграммы

---

## 1. Данные и сущности

**Данные** — информация, описывающая предметную область, хранящаяся в определённой структуре.

**Сущность** — объект реального мира, объединяющий набор связанных данных (например, «Соискатель», «Заявка», «Компания»).

**Атрибут** — конкретная характеристика сущности (например, у сущности «Соискатель»: ФИО, телефон, опыт работы).

---

### Пример сущности и атрибутов:

| Сущность | Атрибуты |
|----------|----------|
| Пользователь | Фамилия, Имя, Отчество, Адрес, Телефон |
| Книга | Название, Автор, ISBN, Год издания |
| Заявка | ID, Название вакансии, Требования, Статус |

---

## 2. Типы атрибутов

| Тип | Описание | Пример |
|-----|----------|--------|
| **Простые** | Базовый тип данных (число, строка, дата) | `salary` (integer) |
| **Составные** | Содержат вложенные атрибуты (JSON, XML) | Хранение шаблона сообщения |
| **Первичный ключ** | Уникально идентифицирует запись | `id`, `phone`, `email` |
| **Внешний ключ** | Ссылается на первичный ключ другой таблицы | `company_id` в таблице заявок |

---

## 3. Ключи

### Первичный ключ
- Уникальный идентификатор записи
- Обязательно должен быть **заполнен** (NOT NULL)
- Не должен изменяться (или меняется крайне редко)
- Пример: `id` пользователя, `табельный номер`

### Внешний ключ
- Ссылается на первичный ключ другой таблицы
- Используется для установления связей между сущностями
- Пример: `applicant_id` в таблице связи с заявками

---

## 4. Связи между сущностями

| Тип связи | Описание | Пример |
|-----------|----------|--------|
| **Один к одному (1:1)** | Одному экземпляру соответствует один | Человек ↔ Паспорт, Автомобиль ↔ VIN-номер |
| **Один ко многим (1:N)** | Одному экземпляру — много других | Одна заявка → много соискателей, Один преподаватель → много лекций |
| **Многие ко многим (N:M)** | Много к многим (реализуется через третью таблицу) | Студенты ↔ Преподаватели, Книги ↔ Авторы |

> **Важно:** Связь «многие ко многим» всегда разводится через **промежуточную (третью) таблицу**, содержащую внешние ключи на обе связываемые сущности.

---

## 5. ER-диаграмма (Entity-Relationship Diagram)

**ER-диаграмма** — это диаграмма, показывающая:
- Какие сущности есть в системе
- Какие атрибуты у каждой сущности
- Как сущности связаны между собой

### Уровни ER-моделей:

| Уровень | Описание | Кто использует |
|---------|----------|----------------|
| **Концептуальный** | Верхнеуровневое представление, только сущности и связи (без атрибутов) | На этапе сбора требований, для оценки объёма работ |
| **Логический** | Добавляются атрибуты, обозначаются типы связей (1:1, 1:N, N:M) | Аналитик для понимания предметной области |
| **Физический** | Привязан к конкретной СУБД (типы данных, названия полей, индексы) | Для передачи разработчикам |

---

### Нотации ER-диаграмм:

| Нотация | Особенности |
|---------|-------------|
| **Чена (Chen)** | Классическая, сущность — прямоугольник, атрибуты — овалы, связи — ромбы |
| **Мартина (Martin)** | Более компактная, атрибуты перечислены внутри сущности |

> **На практике** чаще всего используют **физический уровень** с указанием типов данных и ключей.

## 6. Нормализация данных

**Нормализация** — способ организации данных для устранения избыточности и зависимостей. Всего существует **6 нормальных форм**, но на практике достаточно **первых трёх**.

### Первая нормальная форма (1NF)
- В каждой ячейке таблицы — **одно значение**
- Нет повторяющихся групп
- **Устраняем дублирование данных**

**Пример:** В таблице «Покупки» данные о покупателе дублируются для каждой покупки. Выносим покупателей в отдельную таблицу.

---

### Вторая нормальная форма (2NF)
- Выполнена 1NF
- У каждой записи есть **первичный ключ**
- Все неключевые атрибуты зависят от **всего** первичного ключа (для составных ключей)

**Пример:** Убеждаемся, что у каждой таблицы есть уникальный идентификатор (ID).

---

### Третья нормальная форма (3NF)
- Выполнена 2NF
- Нет **транзитивных зависимостей** (неключевые атрибуты не зависят друг от друга)

**Пример:** В таблице «Сотрудники» поля «Отдел» и «Руководитель отдела» зависят друг от друга. Выносим отделы в отдельную таблицу и связываем через ID.

---

### Денормализация
Намеренное нарушение нормальных форм для:
- **Улучшения производительности** (меньше JOIN-ов)
- **Ускорения чтения** данных

**Пример:** В таблицу пользователей добавляют дублирующую информацию для быстрого получения данных без соединения с другими таблицами.

## 8. Важные практические замечания

### Персональные данные (ФЗ-152)
- Не храните ФИО, телефон, адрес в основной таблице, если это не критично
- Используйте отдельный сервис/таблицу для персональных данных с ограниченным доступом
- **Принцип минимальности:** собирайте только те данные, которые действительно нужны для работы системы

### Именование полей
- Названия сущностей — **в единственном числе** (`applicant`, а не `applicants`)
- Поля с несколькими словами — через **нижнее подчёркивание** (`created_at`, `person_id`)
- Это **рекомендация сообщества**, а не жёсткое требование СУБД, но лучше придерживаться

### Технические vs Бизнесовые атрибуты
- **Бизнесовые** (вверху): статус, зарплата, должность
- **Технические** (внизу): `created_at`, `updated_at`, `id`

### Связующие таблицы
- Иногда техническая таблица для связи «многие ко многим» **превращается в бизнесовую** с собственными атрибутами (статус, комментарий, процент соответствия)
- Это нормальная практика

## 9. Итоговые тезисы (шпаргалка)

1. **Сущность** — объект, **атрибут** — его характеристика
2. **Первичный ключ** уникален, обязателен, неизменяем
3. **Внешний ключ** связывает таблицы
4. Связи бывают: **1:1**, **1:N**, **N:M** (через третью таблицу)
5. **ER-диаграмма** бывает концептуальной, логической и физической
6. **Нормализация** — устранение дублирования данных, 3 первые формы обязательны
7. **Денормализация** — ускорение работы за счёт дублирования (осознанное нарушение)
8. При проектировании **сначала описываем сущности и связи**, затем детализируем атрибуты
9. **Именование:** сущности в единственном числе, поля через `_` (snake_case)
10. **Персональные данные** требуют особого внимания (ФЗ-152)
